2.5.3 Error Identification Module

The error identification module is used to determine whether a generated Japanese translation contains a meaningful translation problem and, if so, what type of problem it is. Rather than assigning a single overall quality label, the system represents error as a structured record consisting of four fields: has_error, severity, categories, and rationale.

Errors are classified using five MQM-style categories: Terminology, Accuracy, Fluency/Grammar, Style/Register, and Locale/Formatting. This structure is useful because it captures not only whether a translation is wrong, but also why it is wrong. In practice, this allows the system to distinguish between, for example, a mistranslated term, an unnatural Japanese sentence, or a formatting mistake such as incorrect punctuation.

The role of error identification differs across the system variants. S0 and S1 do not include a dedicated error-classification module and are therefore evaluated mainly with translation-quality and terminology metrics. S2 includes only a weak automatic error proxy based on prediction-versus-reference mismatch. S3 is the only variant with a true structured error classifier, which predicts whether an error exists, how severe it is, which categories it belongs to, and a short justification. This output is then used both for evaluation and for directing the revision loop inside the agentic pipeline.

3.3 Error Annotation by Gemini

To create a gold reference for error evaluation, we built a labelled error dataset using Gemini as an annotation model. A subset of 500 English-Japanese sentence pairs was first translated automatically to produce candidate system outputs. Gemini was then used as an LLM-as-judge to assess each translation against the English source and assign structured error labels.

For each annotated record, Gemini produced:

  • has_error: whether the translation contains an error
  • severity: none, minor, or major
  • categories: one or more MQM-style error categories
  • rationale: a short explanation of the judgement

The resulting annotations were saved to gemini_annotated_results.jsonl and used as the gold error labels during evaluation. In other words, this file acts as the reference truth against which predicted error labels from the translation pipelines are compared.

4.4 Error Detection

Error detection is evaluated by treating the system’s predicted error label as a classification output and comparing it against the gold label from the Gemini-annotated dataset. This evaluation only applies when both a gold_error_label and a predicted error_check are present for the same row. As a result, error detection metrics are meaningful primarily for S2 and S3, and do not apply to S0 and S1 in the plain baseline setting.

Two metrics are used. Binary F1 measures whether the system correctly identifies the presence or absence of an error. This answers the simplest question: did the system correctly decide whether the translation was acceptable or problematic? Category Macro F1 measures how well the system identifies the type of error across the five MQM-style categories. This is more diagnostic, because it reveals whether the system can tell the difference between terminology problems, meaning distortions, fluency issues, register errors, and locale-formatting mistakes.

In this project, the four variants handle error detection differently:

  • S0 and S1: no dedicated error classifier is used, so error-F1 metrics are not reported.
  • S2: a simple automatic heuristic is used. If the generated Japanese output differs from the reference after normalization, the system marks the translation as having an error. This gives a rough binary signal, but it does not provide reliable category classification.
  • S3: an explicit LLM-based error classifier is used. It reads the English source, the Japanese candidate, and the retrieved context, then predicts structured error JSON containing has_error, severity, categories, and rationale.

This distinction is important when interpreting the results. A low error F1 in S2 does not necessarily mean the translation itself is poor; it may simply reflect the fact that S2 uses a weak proxy for error detection. By contrast, S3’s error metrics are more informative because they come from a genuine structured classifier rather than a simple output mismatch rule.

Plain-English Summary

The project defines error using a fixed gold annotation file and then asks each system to produce its own predicted error label. S0 and S1 do not really do this, S2 does it weakly using a mismatch heuristic, and S3 does it properly using an LLM-based classifier. The predicted label is then compared against the gold label to calculate Binary F1 and Category Macro F1.
